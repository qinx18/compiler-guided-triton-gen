## Multi-Phase Kernel — Split into Separate Triton Kernels

**Both dimensions appear INVALID when the phases are analyzed together**, but that is because different loop bodies (phases) share arrays across iterations.

**Each phase can be parallelized independently.** Split into **separate Triton kernels** — one per top-level `for` loop — and parallelize the spatial dimensions within each kernel. Kernel launch barriers handle cross-phase deps.

For each kernel, use `grid=(cdiv(N, BLOCK),)` with BLOCK >= 128. Within each kernel:
- For phases computing row reductions (e.g., `y[i] += A[i][j] * x[j]`): parallelize rows with `grid=(N,)`, vectorize columns with `tl.arange()`
- For inner reduction loops: use `tl.sum()` over vectorized products
- Use **BLOCK_SIZE >= 128** for column/reduction dimensions
- **Fuse element-wise phases** (e.g., `x[i] = x[i] + z[i]`) into the preceding or following reduction kernel to minimize launch count. Only split when phases have true data dependencies between them.

**For matrix-multiply phases**: Use 2D tiled grid `grid = (cdiv(M, BM), cdiv(N, BN))` with `tl.dot()` accumulation. Each block computes a BM×BN output tile by iterating over the K dimension: `for k in range(0, K, BK): acc += tl.dot(a_tile, b_tile)`.
**IMPORTANT**: The host function MUST call these tiled phase kernels. Do NOT create an additional serial fallback kernel.


## Common Mistakes to AVOID

1. **NEVER use Python for-loops to launch many small kernels.** If you have sequential dims `r, q` and parallel dim `p`, do NOT write `for r in range(NR): for q in range(NQ): kernel[grid](...)`. Instead, fuse them: `grid = (NR * NQ * cdiv(NP, BLOCK),)` and inside the kernel compute `r = pid // (NQ * NP_blocks)`, `q = (pid // NP_blocks) % NQ`, `p_start = (pid % NP_blocks) * BLOCK`.

2. **NEVER write parallel kernels then call a serial fallback.** If you write tiled/vectorized phase kernels (e.g., `phase1_kernel`, `phase2_kernel`), the host function MUST call them. Do NOT add a serial `grid=(1,)` wrapper that ignores the parallel implementations.

3. **Use `tl.dot()` for ALL matrix multiply and outer-product accumulations.** NEVER use scalar triple-nested loops (`for i: for j: for k: acc += a[i,k]*b[k,j]`) inside a Triton kernel. Tile with BLOCK_SIZE and use `acc += tl.dot(a_tile, b_tile)`.

4. **For stencil kernels with `grid=(1,)` guidance: NEVER use multi-block grids.** If the analysis says `grid=(1,)`, do NOT use `grid=(triton.cdiv(N, BLOCK),)`. Multiple CTAs cannot synchronize between phases — CTA 0 may read stale values written by CTA 1. Use `BLOCK = triton.next_power_of_2(N)` so one CTA covers ALL elements.
